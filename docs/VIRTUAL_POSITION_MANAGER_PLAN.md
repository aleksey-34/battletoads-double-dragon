# Path B: Virtual Position Manager (будущее)

## Проблема
Сейчас 1 пара + 1 API-ключ = макс 1 стратегия (Path A валидация).
Для поддержки нескольких стратегий на одну пару нужен слой абстракции.

## Архитектура

### 1. Virtual Positions (per-strategy)
- Новая таблица `virtual_positions`:
  ```sql
  strategy_id INTEGER, symbol TEXT, side TEXT, qty REAL, entry_price REAL
  ```
- Каждая стратегия хранит свой виртуальный размер
- `closeStrategyExposure` закрывает только свою часть (`reduceOnly` на `qty`), а не `closeAll`

### 2. Net Position Reconciler
- Net Position = SUM всех virtual_positions по символу и API-ключу
- Reconciler периодически сравнивает net vs фактическая позиция на бирже
- Deadband (порог нечувствительности): ±5% ИЛИ < min lot — не корректировать
- При расхождении > порога — корректирующий маркет-ордер на дельту

### 3. Защита от flapping
- Фильтр: если |дельта| < min_lot биржи — игнорировать
- Cooldown: не чаще 1 корректировки в N минут на символ
- Логирование всех корректировок в отдельную таблицу

### 4. Защита от задвоения ордеров
- Idempotency key при отправке ордера (clientOrderId)
- Перед открытием — проверка pending orders на бирже
- Если есть открытый ордер с тем же clientOrderId — не отправлять повторно

### 5. Частичное закрытие
- `closeStrategyExposure` принимает `qty` вместо `closeAll`
- Если клиент вручную закрыл часть — reconciler видит расхождение и обновляет virtual_positions

## Приоритет
Низкий. Реализовывать когда появится реальная потребность в нескольких стратегиях на одну пару.
Пока Path A (валидация уникальности) закрывает все кейсы.

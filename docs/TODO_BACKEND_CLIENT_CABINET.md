**Решение**: при формировании `availableSystems` подтягивать `tsBacktestSnapshots[systemName].equityPoints`.

## B. Клиент создаёт свою ТС из офферов

**Новый эндпоинт**: `POST /api/client/strategy/custom-system`
- Принимает: `{ selectedOfferIds, riskLevel, tradeFrequencyLevel, name? }`
- Сохраняет набор как "клиентскую ТС" с меткой `source: 'client_composed'`
- Сохраняет в ту же таблицу офферов / систем, но с пометкой

**Новая таблица или поле**: `custom_trading_systems` или `source` поле в существующей таблице.

**Ограничения**:
- По тарифу: `max_strategies_total` — сколько стратегий можно
- Только 1 ТС на 1 API ключ (не запускать несколько одновременно)

## E. Расширение тарифа (dual product mode)

Сейчас `product_mode` — одно значение на тенанта (`strategy_client` ИЛИ `algofund_client`).
Клиент НЕ может одновременно быть и в стратегиях и в алгофонде.

**Варианты решения**:
1. Сделать `product_mode` массивом: `['strategy_client', 'algofund_client']`
2. Создавать второй тенант с тем же юзером
3. Добавить поле `secondary_product_mode`

**Рекомендация**: вариант 1 — минимальные изменения, одна запись, два потока.
Нужно проверить все `WHERE product_mode =` запросы и адаптировать.

## F. Валидация конфликтов API ключей

✅ **Уже реализовано** в `validateApiKeyNotAssigned()` (service.ts L8968-9030):
- Один ключ не может быть в двух алгофонд-профилях одновременно

## G. Витрина стратегий пуста

Карточки стратегий рисуются из `strategyWorkspace.offers`.
Если витрина пуста — значит в каталоге нет опубликованных офферов для данного тенанта.
**Проверить**: есть ли записи в `strategy_offers WHERE published = 1`.

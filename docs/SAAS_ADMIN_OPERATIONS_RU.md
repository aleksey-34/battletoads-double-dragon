# SaaS Admin: Оферы, ТС, Клиенты, Алгофонд

Практическая инструкция для ежедневной работы админа.

## 1) Блок "Admin обзор / отчеты"

Где: `SaaS -> Admin`, карточка `Admin обзор / отчеты`.

Что проверять:
- `Telegram`: переключатель включен/выключен.
- `token` и `chat_id`: должны быть `ok` для отправки отчета.
- Период отчета: `daily` / `weekly` / `monthly`.

Кнопки:
- `Обновить обзор`: перезагружает performance-отчет по выбранному периоду.
- `Отправить сейчас`: отправляет отчет в Telegram (доступно только если token/chat настроены).

API (для диагностики):
- `GET /api/saas/admin/reports/performance?period=daily|weekly|monthly`
- `GET /api/saas/admin/telegram-controls`
- `PATCH /api/saas/admin/telegram-controls`
- `POST /api/saas/admin/reports/send-telegram`

## 2) Поток по оферам и ТС

Рекомендуемый порядок:
1. `Research`: подготовить/обновить кандидатов.
2. `SaaS -> Admin -> Sweep offers publish/unpublish`: включить оферы в витрину.
3. `Офер и ТС` (бывший раздел `Trading Systems`): собрать/проверить ТС, состав, веса, backtest.
4. `SaaS -> Strategy Client` и `SaaS -> Algofund`: проверить как карточки выглядят для клиента.

Важно:
- Если в разделе `Офер и ТС` пусто, сначала назначьте клиенту API-ключ в `SaaS -> Admin`.
- Период доходности/просадки у оферов должен читаться как `Xd • dateFrom -> dateTo`.
- График equity в админской таблице оферов открывается через раскрытие строки.

## 3) Аппрув и снятие с витрины

При публикации:
- Проверить метрики (`Ret`, `DD`, `PF`, `TPD`) и период теста.
- Проверить equity-кривую (раскрытая строка офера).

При снятии с витрины:
- Проверить, есть ли подключенные клиенты к оферу/ТС.
- Проверить, есть ли открытые позиции у затронутых ТС.
- Если есть позиции: предложить действие `остановить/закрыть позиции` или `отмена`.

Примечание: автоматический мастер снятия с витрины с обязательной проверкой позиций пока не выделен в отдельный wizard; делать через операционный чеклист.

## 4) Клиенты: массовые и точечные действия

Текущий процесс:
- Массовые операции и продвинутый bulk-поток (чекбоксы + пакетный switch/start/stop) пока не выделены отдельной таблицей.
- Базовый рабочий контур идет через очередь запросов Algofund и админ-аппрув.

Целевой UX (план):
- Таблица `Подключенные клиенты` с чекбоксами.
- Фильтр действия: `Офер`/`ТС`.
- Для `Офер`: выбор типа стратегии.
- Для `ТС`: выбор по имени ТС.
- Batch-кнопки: `Start`, `Stop`, `Switch TS`.

## 5) Как остановить/запустить/переключить Алгофонд у клиента

Модель уже поддерживается через request queue.

Шаги:
1. Создать запрос клиента:
   - `POST /api/saas/algofund/:tenantId/request`
   - `requestType`: `start` | `stop` | `switch_system`
   - Для `switch_system`: передать `targetSystemId` (и опционально `targetSystemName`).
2. Админ обрабатывает запрос в очереди:
   - `POST /api/saas/algofund/requests/:requestId/resolve`
   - `status`: `approved` или `rejected`
   - `decisionNote`: комментарий решения.

Что делает approve:
- `start`: включает профиль/материализацию.
- `stop`: выключает профиль.
- `switch_system`: переключает на выбранную ТС.

## 6) Предложенная структура админки (целевая)

`Админ`
- `Офер и ТС`
  - карточки с витрины (strategy + algofund)
  - редактирование параметров
  - backtest/equity
  - publish/unpublish с проверкой подключений/позиций
- `Анализ ресерча`
  - новые карточки из research
  - пометка "свежее из ресерча"
  - передача в `Офер и ТС` на аппрув
- `Клиенты`
  - подключенные клиенты
  - массовые действия
  - запросы на подключение
  - тарифные сетки
- `Мониторинг`
  - отчеты по оферам/ТС
  - low-lot
  - системные алерты

## 7) Ближайший порядок внедрения

1. Стабилизировать IA и названия разделов (`Офер и ТС`, `Клиенты`, `Анализ ресерча`, `Мониторинг`).
2. Вынести общий `Кабинет клиента` из вкладок продукта в единый блок админ-навигации.
3. Добавить `Подключенные клиенты` с checkbox + batch-actions.
4. Добавить guardrail при снятии офера с витрины (клиенты + открытые позиции + предложенные действия).

## 8) Фиксация модели ТС и тарифной сетки

### Модель
- Витринные карточки (`Offer`) — это источник конфигурации и классификации.
- `Trading System` у клиента/tenant — runtime-материализация выбранных карточек и разрешенных настроек.
- TS не должны концептуально классифицироваться только по API-ключу; API-ключ — это контур исполнения, а не бизнес-классификация.

### Лимиты Strategy Client (зафиксировано)
- Ограничение для сборки **одной** пользовательской ТС: от 2 до 6 офферов (по тарифу).
- Ограничение на количество пользовательских ТС: от 1 до 3 (по тарифу).

Сетка:

| План | Офферов в 1 ТС | Пользовательских ТС |
| --- | --- | --- |
| strategy_15 | 2..2 | 1 |
| strategy_20 | 2..3 | 1 |
| strategy_25 | 2..4 | 2 |
| strategy_30 | 2..4 | 2 |
| strategy_50 | 2..5 | 3 |
| strategy_100 | 2..6 | 3 |

Примечание:
- Текущий runtime-профиль хранит один активный собранный набор офферов; лимит "до 3 ТС" уже зафиксирован в тарифных ограничениях и UI-констрейнтах как целевая бизнес-норма для следующего шага с multi-TS профилями.

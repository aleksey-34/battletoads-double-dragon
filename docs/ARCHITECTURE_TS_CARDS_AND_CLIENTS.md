# Архитектура управления карточками ТС, офферами и пользовательскими настройками

## 1. Мастер-карточка ТС (Trading System Master Card)
- **Единая сущность** для каждой торговой системы или оффера.
- Содержит все базовые параметры: состав участников, веса, режимы, лимиты, параметры риска и частоты сделок.
- Публикуется и управляется только админом.
- Все клиенты подключаются к одной и той же мастер-карточке.

## 2. Подключение клиента (Client Attachment)
- Для каждого клиента создаётся запись о подключении к мастер-карточке ТС.
- В этой записи хранятся только индивидуальные настройки клиента:
  - Индивидуальный риск-модификатор
  - Индивидуальная частота сделок (если разрешено тарифом)
  - Прочие пользовательские параметры (например, лимиты, API-ключи)
- **Изменения в мастер-карточке** автоматически применяются ко всем клиентам, если они не переопределили конкретный параметр.

## 3. Логика исполнения и отображения
- **Витрина** всегда отражает мастер-карточку (актуальные параметры, состав, метрики).
- **Статистика клиента** строится на основе мастер-карточки + его индивидуальных модификаторов.
- **Изменения админом** в мастер-карточке (например, смена состава, лимитов) сразу видны всем клиентам.
- **UI для клиента** — только свои настройки, только своя статистика, без доступа к админским бэктестам.

## 4. База данных
- Таблица `trading_systems` (master-карточки)
- Таблица `client_attachments` (подключения клиентов к карточкам)
- Таблица `user_settings` (индивидуальные параметры)
- Все связи через foreign key, нет дублирования ТС на каждого клиента.

## 5. Контроль целостности
- При старте рантайма — проверка наличия всех нужных мастер-карточек.
- Если база пуста — критическая ошибка, торговля не запускается.
- Регулярные бэкапы БД.

## 6. Восстановление и миграции
- Скрипты для инициализации структуры БД и импорта карточек из файлов.
- Возможность массового обновления карточек и их параметров через админку или API.

## 7. Рефакторинг и ветка в git
- Все изменения по архитектуре и управлению карточками — в отдельной ветке (например, `feature/ts-architecture-refactor`).
- Каждый шаг фиксируется отдельным коммитом с подробным описанием.
- Вопросы и спорные моменты выносятся в issues/discussions.
